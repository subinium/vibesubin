# vibesubin

여러분의 AI 어시스턴트가 리팩토링하고, 보안을 살피고, 배포를 세팅할 때 제가 바라는 방식대로 동작하도록 가르치는 포터블 스킬 플러그인입니다. 그리고 그 모든 것을 한 단어로 한꺼번에 돌릴 수 있습니다 — `/vibesubin`.

개발자로 훈련받지는 않았지만 AI 와 함께 실제 프로젝트를 굴리고 계신 분들을 위해 만들었습니다. 제가 AI 와 함께 코딩할 때 쓰는 습관들을 스킬로 묶어 둔 것입니다. 그래서 사용자가 규칙 하나하나를 알고 있지 않아도, AI 가 그 규칙들을 따라 움직입니다. 같은 스킬이 지루한 규율을 손으로 반복하기 싫은 다른 개발자에게도 쓸모가 있습니다.

동일한 `SKILL.md` 파일이 세 가지 설치 경로를 구동합니다: **Claude Code** (플러그인 마켓플레이스 경유), **Codex CLI** (`--to codex` 심볼릭 링크 경유), 그리고 **[skills.sh](https://skills.sh)** 가 지원하는 모든 에이전트 (Cursor, Copilot, Cline 등). 에이전트별 포크는 없습니다. 한 번 쓴 스킬이 사용하시는 모든 호스트에 그대로 올라갑니다.

## 빠른 시작

[Claude Code](https://code.claude.com) 를 설치하신 다음, Claude Code 세션에서 아래를 실행하십시오:

```
/plugin marketplace add subinium/vibesubin
/plugin install vibesubin@vibesubin
```

아무 레포든 열고 `/vibesubin` 을 입력하십시오. 플러그인이 제공하는 모든 스킬이 여러분의 코드를 상대로 병렬로, 읽기 전용으로 돌아갑니다. 그리고 하나의 우선순위 리포트가 돌아옵니다 — 스페셜리스트별 신호등, 파일 단위로 묶인 상위 10 개 수정 리스트, 권장 작업 순서, 그리고 한 문장 판정입니다. 사용자가 리스트에서 항목을 승인하기 전까지 어떤 파일도 수정되지 않습니다.

Claude Code 가 아니라 Codex CLI, Cursor, Copilot, Cline 을 쓰신다면, 크로스 에이전트 설치인 [경로 B](#경로-b--skillssh-크로스-에이전트) 나 수동 설치인 [경로 C](#경로-c--수동-심볼릭-링크-claude-code--codex-cli) 로 바로 넘어가시면 됩니다. 세 경로 모두 아래 [설치](#설치) 섹션에서 자세히 다룹니다.

## vibesubin 이 무엇이고 무엇이 아닌가

짧게 말하자면, 이것은 `SKILL.md` 파일들로 작성된 작은 AI 스킬 묶음입니다. 사용자의 요청이 그중 하나에 맞을 때마다 에이전트가 자동으로 집어서 실행합니다. 트리거 문구를 외우실 필요는 없습니다. 평소대로 말씀하시면 됩니다 — *"이 파일 안전하게 쪼개 줘"*, *"뭐 새는 거 있어?"*, *"배포 세팅해 줘"* — 알맞은 스킬이 알아서 발동합니다.

이 스킬들을 하나로 묶을 만한 가치가 있는 이유는 공통된 규칙 하나 때문입니다: **근거를 보여줄 수 있기 전까지 "완료" 라고 말하지 않습니다.** 리팩토링은 AI 가 파일을 다시 썼다고 완료되는 것이 아닙니다. 네 개의 독립된 검사가 누락·이동·연결 오류가 없다고 확인해야 완료입니다. 보안 스윕은 *"괜찮아 보입니다"* 같은 분위기 기반 단락이 아닙니다. 각 히트가 실제 문제인지, 오탐인지, 사람 검토가 필요한지로 분류되고, 파일명과 라인 번호가 함께 붙는 우선순위 리스트입니다. 스킬이 반환할 때, 주장하는 모든 내용은 사용자가 직접 실행해 볼 수 있는 커맨드로 뒷받침됩니다.

그리고 일찍 알고 계시면 좋은 것이 이 플러그인이 **무엇이 아닌가** 입니다. SaaS 대시보드가 아닙니다 — 서버가 없으며, 사용자의 AI 가 직접 보내는 것 외에는 어떤 것도 이 머신을 떠나지 않습니다. 컴플라이언스 도구도 아닙니다 — SOC 2 나 HIPAA 감사가 필요하시다면 그 목적으로 만든 도구를 쓰셔야 합니다. 그리고 코드 생성기나 프로젝트 스캐폴드도 아닙니다. 이미 있는 레포를 개선하는 도구이며, 범위를 벗어난 질문에는 솔직하게 그렇다고 말합니다.

### 현재 라인업

## `/vibesubin` 명령어

`/vibesubin` 은 두 가지를 동시에 담고 있으며, 둘 다 의도된 것입니다.

명령어입니다 — 한 단어로 플러그인 안의 모든 스킬을 레포에 병렬로 돌리고, 결과를 하나의 리포트로 합치고, 수정이 적용되기 전에 사용자의 승인을 기다립니다. 스킬을 고르실 필요도, 직접 체인을 만드실 필요도, 스킬이 차례로 돌아가는 동안 오후를 기다리실 필요도 없습니다. 그냥 단어를 입력하시면 됩니다.

그리고 하나의 바이브이기도 합니다. 이 플러그인은 작업 방식 하나를 이름으로 가집니다 — *바이브 코딩*, 매 줄을 직접 쓰지 않고 AI 와 함께 뭔가 진짜를 만드는 방식 — 그리고 `/vibesubin` 은 매 단계를 서술하지 않고 전체 점검을 받고 싶을 때 입력하시는 단어입니다. 반은 명령어, 반은 밈입니다. 의도된 것입니다.

실제로 실행하면 벌어지는 일은 이렇습니다. 스킬이 한 문장으로 무엇을 할 것인지 말하고, 범위를 좁힐 기회를 드립니다. `src/api/` 만 보고 싶으시거나, 보안과 repo rot 만 확인하고 싶으시면 실행 전에 말씀하시면 됩니다. 그렇지 않으면 모든 스페셜리스트가 병렬 task agent 로 펼쳐집니다. 스페셜리스트들은 서로 상태를 공유하지 않습니다 — 오케스트레이션 LLM 셋업을 써 보셨다면 에이전트 사이의 교차 오염이 흔한 실패 모드라는 걸 아실 것입니다. `/vibesubin` 은 각 실행을 격리해서 이를 피합니다.

이 단계의 스페셜리스트들은 전부 읽기 전용입니다. 발견을 만들지 수정은 만들지 않습니다. 특정 스페셜리스트가 실행될 수 없는 경우 — 예컨대 테스트 슈트가 시작도 안 돼서 `refactor-safely` 가 그린 베이스라인을 잡을 수 없는 경우 — 그 사실 자체를 발견으로 보고하고 길을 비킵니다. 다른 스페셜리스트들은 계속 진행합니다. 하나의 스페셜리스트 실패 때문에 스윕 전체가 막히는 일은 없습니다.

모두 반환되면 결과가 합성 규칙에 따라 병합됩니다. 카테고리와 무관하게 크리티컬이 맨 위로 올라옵니다 — 유출된 시크릿이 핫스팟을, 핫스팟이 누락된 docstring 을 이깁니다. 두세 스페셜리스트가 독립적으로 같은 파일을 지목하면 그 파일의 우선순위가 뛰어오릅니다. 핫스팟이면서, SQL 인젝션이 있고, 테스트 커버리지가 0 인 파일은 가장 먼저 고쳐야 할 파일이며, 리포트가 그 사실을 말합니다. 발견은 파일 단위로 묶입니다. 사용자가 고치는 건 카테고리가 아니라 파일이기 때문입니다. 그리고 모든 항목에는 구체적인 권고 하나와 그걸 처리할 스킬 포인터가 붙습니다.

마지막에 여러분이 보시는 것은 하나의 마크다운 리포트입니다: 전체 *바이브* 를 요약한 한 단락, 스페셜리스트별 신호등 한 줄, 상위 10 개 수정 리스트 표, 수정을 처리할 권장 순서, 한 문장 판정. 모든 것이 괜찮다면 판정이 그렇게 말합니다 — *"깨끗해 보입니다, 폴리시 두 개 있습니다"* 가 정당한 출력입니다. 여러분이 어떤 항목을 적용할지 말씀하시기 전까지 레포의 어떤 것도 바뀌지 않습니다.

### 리포트 모양

모든 실행이 동일한 섹션을 만듭니다. 아래 골격은 템플릿이며, 실제 내용은 레포를 상대로 돌아간 스페셜리스트들이 만들어 냅니다.

```markdown
# /vibesubin sweep — <레포명> — <날짜>

## Vibe check

<한 단락. 레포의 전반적인 상태, 눈에 띄는 두세 가지, 그리고 배포해도 될지
아니면 먼저 고쳐야 할지. 응원도 없고 어물쩍 넘기지도 않습니다.>

## What ran

- refactor-safely:    🟢 | 🟡 | 🔴   <한 줄 요약>
- audit-security:     🟢 | 🟡 | 🔴   <한 줄 요약>
- fight-repo-rot:     🟢 | 🟡 | 🔴   <한 줄 요약>
- write-for-ai:       🟢 | 🟡 | 🔴   <한 줄 요약>
- setup-ci:           🟢 | 🟡 | 🔴   <한 줄 요약>
- manage-config-env:  🟢 | 🟡 | 🔴   <한 줄 요약>
  <... 이후 추가된 스킬 ...>

## Top-10 fix list

| # | 파일 | 무엇이 잘못됐는가 | 심각도 | 담당 스킬 |
|---|---|---|---|---|
| 1 | <파일:라인> | <발견> | CRITICAL / HIGH / MEDIUM / LOW | <스킬 → 스킬> |

## Recommended order

1. <왜 이것부터>
2. <왜 이게 다음>

## What I did NOT do

읽기 전용 스윕이었습니다. 파일은 바뀌지 않았고, 커밋도 만들어지지 않았으며,
외부 시스템도 건드리지 않았습니다. 항목을 승인하시면 각각이 담당 스페셜리스트로
핸드오프됩니다.

## Verdict

<한 문장. 직설적이며, 어물쩍 넘기지 않습니다.>
```

스킬이 따르는 합성 규칙을 포함한 전체 템플릿은 umbrella [`SKILL.md`](./plugins/vibesubin/skills/vibesubin/SKILL.md) 안에 들어 있습니다. 스킬이 실제로 무엇을 만들어 내도록 지시받는지 단어 단위로 확인하고 싶으시면 그 파일을 여십시오.

### 스윕을 돌릴 때와 스킬을 직접 부를 때

스윕은 열린 질문에 잘 맞습니다. *"레포를 방금 넘겨받았는데 상태가 어떻습니까?"*, *"이거 배포해도 괜찮습니까?"*, *"이거 건드리기 전에 두 번째 의견이 필요합니다."* 이런 것들이 `/vibesubin` 질문입니다 — 무엇을 찾아야 할지 모르시고, 플러그인이 알려주기를 원하시는 경우입니다.

반면 원하시는 것이 이미 분명하다면 스킬을 직접 부르십시오. *"이 한 파일만 리팩토링"* 은 `/refactor-safely` 입니다. *"실수로 `.env` 를 GitHub 에 푸시했습니다"* 는 지금 당장 `/audit-security` 입니다 — 급한 건은 속도가 범위를 이깁니다. *"README 를 써 주십시오"* 는 `/write-for-ai` 입니다. 대상이 이미 분명한데 전체 스윕을 돌리는 것은 자신을 더 오래 기다리게 만들고 더 긴 리포트를 훑게 만들 뿐입니다.

### 알아두시면 좋은 엣지 케이스 몇 가지

스페셜리스트가 실행 도중 실패하면 나머지는 계속 돌고, 실패한 스페셜리스트는 *What ran* 섹션에 🔴 로 표시되고 오류가 인라인으로 인용됩니다. 어느 한 스페셜리스트의 실패 때문에 스윕이 막히는 일은 없습니다.

레포가 커서 (파일 만 개 이상) 하나의 스윕으로 처리하기 어려우면, `/vibesubin` 이 최상위 서브디렉토리 단위로 파티셔닝해서 각 파티션을 돌리고 리포트를 병합합니다. 시작 전에 *파티셔닝* 알림이 뜨므로 왜 오래 걸리는지 아실 수 있습니다.

모든 스페셜리스트가 깨끗하게 반환되면 판정이 그렇게 말합니다. 명령어를 정당화하려고 가짜 수정 리스트가 만들어지지 않습니다. *"괜찮습니다. 접힘선 아래의 노이즈는 무시하셔도 됩니다"* 가 진짜 결과입니다.

## 설치

동일한 스킬 파일, 세 가지 설치 경로입니다. 사용하시는 에이전트와 업데이트를 어떻게 받고 싶으신지에 따라 고르십시오. 경로들은 공존할 수 있습니다 — Claude Code 용으로 경로 A 를 설치해 두고 동시에 Codex CLI 용으로 경로 C 를 걸어도 아무 문제없습니다.

| 경로 | 대상 에이전트 | 이럴 때 좋습니다 |
|---|---|---|
| **A. Claude Code 마켓플레이스** | Claude Code | 자동 업데이트와 가장 단순한 흐름을 원하실 때 |
| **B. `skills.sh` (npx)** | Claude Code, Cursor, Copilot, Cline, Codex CLI 등 | Claude Code 가 아닌 다른 에이전트를 쓰시거나 여러 에이전트를 쓰실 때 |
| **C. 수동 심볼릭 링크** | Claude Code / Codex CLI | 플러그인 자체를 수정하시거나 `git pull` 이 즉시 반영되기를 원하실 때 |

### 경로 A — Claude Code 플러그인 마켓플레이스 (권장)

먼저 [Claude Code](https://code.claude.com) 가 설치되어 있어야 합니다. 그다음 마켓플레이스를 한 번 등록하십시오:

```
/plugin marketplace add subinium/vibesubin
```

플러그인을 설치하십시오:

```
/plugin install vibesubin@vibesubin
```

커맨드 안의 두 `vibesubin` 은 플러그인 이름과 마켓플레이스 이름입니다. 의도적으로 동일합니다 — 단일 플러그인 마켓플레이스이기 때문입니다.

설치가 됐는지 확인하려면, 새 Claude Code 세션에서 `/` 를 누르고 `refactor` 를 입력해 보십시오. 자동완성에 `/refactor-safely` 가 다른 스킬들과 함께 떠야 합니다. 뜨면 끝난 것입니다.

업데이트가 필요하실 때는 `/plugin marketplace update` 를 실행하십시오. 몇 주에 한 번, 혹은 GitHub 에서 새 릴리스 안내를 확인하셨을 때 돌리시면 됩니다. 마켓플레이스 업데이트는 여러분이 설치해 둔 다른 것을 건드리지 않습니다.

잘못될 수 있는 몇 가지: *"repository not found"* 가 뜨면 철자를 확인하십시오 — `subinium/vibesubin`, 소문자, 슬래시 하나입니다. 자동완성에 스킬이 안 보이면 Claude Code 세션을 전부 닫고 (세션 단위로 캐싱됩니다) 새 세션을 여십시오. 마켓플레이스 등록이 *"already exists"* 라고 하면 이미 등록된 것이므로 `/plugin marketplace update` 로 넘어가시면 됩니다.

### 경로 B — `skills.sh` (크로스 에이전트)

[`skills.sh`](https://skills.sh) 는 Vercel Labs 가 운영하는 크로스 에이전트 스킬 레지스트리입니다. 그 CLI 는 vibesubin 이 쓰는 동일한 `SKILL.md` 포맷을 읽습니다. 즉, `skills` CLI 가 지원하는 모든 호스트 — Claude Code, Cursor, GitHub Copilot, Cline, Codex CLI 등 — 에서 이 플러그인이 그대로 동작합니다.

한 줄로 모든 것이 설치됩니다:

```bash
npx skills add subinium/vibesubin
```

내부적으로 `skills` 는 레포를 클론하고, `plugins/vibesubin/skills/` 를 순회하며 유효한 YAML frontmatter 를 가진 `SKILL.md` 파일을 찾고, 각각을 활성 에이전트의 skills 디렉토리에 설치합니다. Claude Code 는 필요하지 않습니다.

업데이트는 같은 명령을 다시 실행하시면 됩니다. 제거는 `npx skills remove vibesubin` 입니다.

`skills` 가 스킬을 찾지 못하면 사용 중인 에이전트가 아직 지원 목록에 없을 수 있습니다 — `npx skills --help` 가 현재 타겟을 보여줍니다. 다른 에이전트가 타겟으로 잡힌다면 같은 help 출력에 오버라이드 플래그가 있습니다. 그리고 `npx` 가 오래된 버전을 캐싱하고 있다면 `--force` 나 `npx clear-npx-cache` 로 보통 해결됩니다.

### 경로 C — 수동 심볼릭 링크 (Claude Code / Codex CLI)

플러그인 자체를 수정하시거나, 오프라인 설치가 필요하시거나, `git pull` 만으로 변경 사항이 재설치 없이 세션에 즉시 반영되기를 원하실 때 이 경로를 쓰십시오. `install.sh` 는 각 스킬 디렉토리를 대상 에이전트의 skills 폴더에 심볼릭 링크로 걸어 둡니다. 복사할 것도, 새로고침할 것도 없습니다.

```bash
cd ~/Projects            # 또는 코드를 두시는 곳
git clone https://github.com/subinium/vibesubin.git
cd vibesubin

bash install.sh                  # 기본값: Claude Code (~/.claude/skills)
bash install.sh --to codex       # Codex CLI (~/.codex/skills)
bash install.sh --to all         # 양쪽에 한 번에 symlink
bash install.sh --dry-run        # 실제 변경 없이 미리보기
bash install.sh --force          # 이름 충돌 덮어쓰기
```

확인 방법은 다른 경로와 동일합니다: 에이전트를 시작하고, `/` 를 누르고, 스킬을 찾으시면 됩니다. Claude Code 는 자동완성에 띄우고, Codex 는 커맨드 메뉴에 나열합니다.

업데이트는 클론에서 이루어집니다. `git pull` 이 소스를 갱신하면 대상 디렉토리가 심볼릭 링크이므로 새 파일이 다음 세션에서 바로 반영됩니다. 재설치도, 재등록도 필요 없습니다.

제거는 설치 플래그를 그대로 따릅니다:

```bash
bash uninstall.sh                # Claude Code
bash uninstall.sh --to codex     # Codex CLI
bash uninstall.sh --to all       # 양쪽 모두
```

제거 스크립트는 자기가 만든 심볼릭 링크만 지웁니다. 대상 디렉토리에 실제 파일이나 다른 곳을 가리키는 심볼릭 링크가 있다면 그대로 둡니다. 여러분의 클론은 절대 건드리지 않습니다.

`install.sh` 가 *"Permission denied"* 를 던지면 `chmod +x install.sh` 를 실행하고 다시 시도하십시오. 이름 충돌이 나면 `--dry-run` 이 무엇이 걸리는지 보여주고 `--force` 가 덮어씁니다. 실수로 경로 A 와 경로 C 로 중복 설치하셨다면, 클론에서 `bash uninstall.sh --to all` 을 돌려 심볼릭 링크를 정리한 뒤 유지하실 경로 하나를 선택하십시오.

Codex CLI 관련해서 한 가지 말씀드립니다: 사용자 레벨 skills 디렉토리가 `~/.codex/skills/` 라고 가정하고 있습니다. Codex 가 사용자 설정에 `~/.codex/` 를 쓰는 컨벤션에 기반한 추정입니다. 만약 여러분의 Codex 버전이 다른 경로를 쓴다면, 버전과 실제 경로를 적어 이슈를 열어 주십시오. 스크립트가 업데이트됩니다. 그동안에는 `install.sh` 의 타겟을 패치하시거나 클론을 직접 symlink 하시면 됩니다.

## 스킬

각 스킬은 [`plugins/vibesubin/skills/`](./plugins/vibesubin/skills/) 아래의 디렉토리입니다. 아래 설명은 사용자 대상 버전이며, 전체 방법론은 각 스킬의 `SKILL.md` 에, 더 깊은 레퍼런스는 인접한 `references/` 에 들어 있습니다. 스킬 파일을 직접 읽으실 필요는 없습니다 — AI 가 읽습니다 — 다만 오픈소스이므로 궁금하시면 여실 수 있습니다.

아래 목록은 현재 릴리스 기준이며, 계속 자랍니다. 새 스킬은 동일한 규칙을 물려받고 `/vibesubin` 이 자동으로 픽업합니다.

### 1. `refactor-safely`

이 플러그인에서 제가 가장 강하게 변호할 스킬이 있다면 이것입니다. AI 가 코드를 건드릴 때 가장 큰 실패 모드는 조용한 실패입니다 — 함수가 옮겨졌고, 이름 변경이 성공했고, 테스트도 통과했는데, 3 주 뒤 누군가가 옛 이름을 여전히 가리키는 코드 경로를 건드리자 전체가 터지는 경우. `refactor-safely` 는 바로 이런 종류의 조용한 실패를 불가능하게 만들기 위해 존재합니다.

동작 방식은 이렇습니다. 무엇을 건드리기 전에 레포의 현재 상태를 스냅샷합니다. 어떤 함수가 있는지, 어떤 클래스가 있는지, 각각이 어느 파일에 있는지, 지금 테스트가 통과하는지, 린터가 뭐라 하는지, 스모크 실행이 무엇을 출력하는지. 그것이 *이전* 스냅샷입니다. 그다음 변경을 의존성 트리로 분해합니다 — *"이 파일을 쪼개려면 먼저 이 헬퍼를 추출하고, 헬퍼를 추출하려면 먼저 저 변수 이름을 바꾼다"* — 그리고 리프에서 루트 방향으로 작업합니다. 실행 도중 레포가 반쯤 리팩토링된 상태에 빠지지 않습니다.

각 노드 이후에 네 개의 독립적인 검증이 돌아갑니다. 첫 번째는 심볼 집합을 순회해서 변경 전에 있던 모든 함수·클래스·export 이름이 여전히 어딘가에 존재하는지 확인합니다 — 조용한 누락을 잡아냅니다. 두 번째는 옮겨진 함수 본문에 대해 AST diff 를 돌려서 공백과 주석을 제외하면 라인 단위로 이전과 동일한지 확인합니다. 세 번째는 테스트, 타입체크, 린트, 스모크 실행을 다시 돌려서 코드가 하던 일을 여전히 하는지 확인합니다. 네 번째가 — 실제로 가장 많은 버그를 잡아내는 패스입니다 — 옮겨진 모든 심볼의 모든 호출부를 걸어보고 새 위치를 가리키는지 확인합니다. 넷 중 하나라도 실패하면 스킬은 전진하지 않습니다. 원인을 고치고 같은 노드에서 다시 돌립니다. 전부 그린이 될 때까지.

`refactor-safely` 가 하지 않는 것들 — 알고 계시면 좋습니다:

- 사용자가 지시하지 않은 파일을 건드리는 일.
- 네 개의 검증 중 하나라도 실패한 상태에서 *"완료"* 라고 말하는 일. 원인을 고치거나, 아니면 멈추고 이유를 말합니다.
- 테스트 결과를 지어내는 일. 의존성이 빠졌거나 환경이 깨져서 테스트 슈트가 시작할 수 없다면, 그 사실을 말하고 멈추지, 추측하지 않습니다.
- *이동* 작업 중 함수 본문을 다시 쓰는 일. 이동은 비트 단위 복사이며, 스킬이 그걸 명시적으로 검증합니다.

이 스킬이 막아주는 함정을 한 문장으로 요약하자면: 깨끗하게 빌드되고 테스트도 통과하지만 아무도 테스트로 커버하지 않은 한 코드 경로를 부수는 리팩토링입니다. Call-site closure 검사가 그걸 잡아냅니다.

### 2. `audit-security`

엔터프라이즈 보안 스캐너에는 노이즈 문제가 있습니다. 500 개를 지적하고 그중 490 개가 오탐이면, 일주일 뒤에 비개발자는 전체를 무시하게 됩니다. 진짜 크리티컬이 묻혀 사라집니다. `audit-security` 는 의도적으로 그 반대 모양입니다: 실제로 사람이 저지르는 실수를 잡는 짧고 손으로 고른 패턴 리스트, 그리고 모든 히트를 분류한 결과 — 실제 문제, 오탐, 또는 사람 검토 필요 — 에 한 줄 이유가 붙습니다.

뻔한 것들 (커밋된 시크릿, 문자열 이어 붙이기로 만든 SQL, 같은 방식으로 만든 셸 커맨드, 신뢰할 수 없는 입력에 대한 `eval`/`exec`/`pickle.loads`) 을 확인하고, 덜 뻔한 것들 (파일 읽기에 들어가는 사용자 제어 경로, 이스케이프 없이 박힌 HTML 의 사용자 입력, `httpOnly` 나 `Secure` 가 빠진 쿠키, 와일드카드 CORS) 도 확인하고, 사람들이 자주 잊는 것들 (작업 트리뿐 아니라 깃 히스토리에 있는 `.env` 파일, `.pem` 과 SSH 키) 까지 확인합니다. 실제 발견마다 평이한 말로 심각도가 붙습니다 — *"낯선 사람이 모든 사용자 데이터를 읽을 수 있습니다"* 가 *"CWE-89"* 보다 비개발자에게 낫습니다 — 그리고 한 문장짜리 수정안과 그걸 적용할 스킬 포인터가 함께 붙습니다.

펜테스트를 대체하지는 않습니다. 실행 중인 시스템을 탐색하지 않고, 인증된 스캔도 하지 않고, 네트워크를 전혀 건드리지 않습니다. 레포에 대한 정적 스윕입니다. 그리고 난처한 것이라고 해서 *"괜찮아 보입니다"* 요약 뒤에 숨기지 않습니다. 크리티컬이 있다면 여러분이 좋아하시든 말든 맨 위에 있습니다.

이 스킬이 막아주는 함정은 아무도 출력을 읽지 않는 상황입니다. 실제로 읽는 10 줄짜리 분류된 리스트가 읽지 않는 500 줄짜리 PDF 보다 가치가 큽니다.

### 3. `fight-repo-rot`

레포는 하룻밤에 무너지지 않습니다. 갓 파일이 자라고, 죽은 코드가 쌓이고, 하드코딩된 경로가 커밋되고, 6 개월 된 `TODO` 가 영원한 `TODO` 가 되면서 몇 달에 걸쳐 썩어갑니다. 보통 뭔가가 깨진 뒤에야 썩음을 눈치챕니다. `fight-repo-rot` 은 "있었으면 좋았을" 조기 경보 리포트입니다.

주 지표는 많은 개발자가 결국 힘들게 배우는 것입니다: **churn × complexity**. 자주 바뀌면서 *동시에* 복잡한 파일이 다음에 여러분이 깨뜨릴 확률이 가장 높은 파일입니다. 그런 파일들이 리스트 맨 위에 올라옵니다. 그 외에도 스킬이 갓 파일 (500 줄 초과), 갓 함수 (50 줄 초과), 죽은 코드 (아무도 부르지 않는 함수, 아무도 임포트하지 않는 파일), 하드코딩된 절대 경로와 IP 리터럴, 6 개월 넘은 `TODO` / `FIXME` / `HACK` 주석, 그리고 모두가 임포트하는 파일 (훌륭한 리팩토링 대상) 이나 모두를 임포트하는 파일 (결합도 악취) 을 스캔합니다.

돌아오는 결과는 *"이것부터 고치십시오"* 순으로 정렬된 표입니다. 각 항목에는 처리할 스킬 포인터가 붙습니다. 이 스킬은 진단이지 치료가 아닙니다 — 리팩토링하지도, 다시 쓰지도, 삭제하지도 않습니다. 단지 어디를 봐야 할지 말해 줍니다. 그래서 여러분의 리팩토링 노력이 가장 최근에 열어 본 파일이 아니라 올바른 대상에 떨어집니다.

### 4. `write-for-ai`

대부분의 문서는 한 번 훑는 사람 독자를 위해 작성됩니다. AI 는 다르게 읽습니다. 매 세션마다 처음부터 다시 파싱하고, 산문보다 표를, 애매한 참조보다 백틱으로 감싼 명시적인 파일 경로를, 이야기 속에 묻힌 규칙보다 선언적으로 적힌 *"X 하지 마십시오"* 를 선호합니다. `write-for-ai` 는 그런 독자를 위해 씁니다 — 그리고 결과물은 사람에게도 더 잘 먹힙니다. 구조는 구조이기 때문입니다.

스킬에게 쓸 것을 주시면 — README, 커밋 메시지, PR 설명, 아키텍처 문서, `CLAUDE.md`, `AGENTS.md` — 새 AI 세션이 그러하듯 레포를 처음 보듯 읽는 것부터 시작합니다. 패키지 매니페스트, 설정 파일, 엔트리 포인트, 디렉토리 구조, 방금 도착한 사람이라면 먼저 볼 것들. 그다음 불변식을 추출합니다: 프로젝트가 *무엇을* 하는지뿐 아니라 어떤 규칙을 따르는지 (브랜치가 어떻게 돌아가는지, 시크릿은 어디 사는지, 테스트를 돌리는 커맨드는 무엇인지, *"완료"* 의 정의는 무엇인지). 해당 템플릿을 채웁니다. 그리고 — 이게 중요한 부분입니다 — 쓰기 전에 문서가 하는 모든 주장을 검증합니다. README 가 *"`pnpm test` 가 슈트를 돌립니다"* 라고 쓴다면, 스킬이 먼저 `pnpm test` 를 실행해 봅니다. 동작하지 않으면 문서는 쓰여지기 전에 현실에 맞춰 수정됩니다.

이 스킬이 실제로 막아주는 것: AI 에게 README 를 다시 써 달라고 하면 잘 씁니다. 그 과정에서 환경변수에 대한 단락이 조용히 사라집니다. 이전 버전에 뭐가 있었는지 기억이 안 나서 눈치채지 못합니다. 다음 AI 세션이 새 README 에서 시작하고, 여러분의 env 레이아웃이 존재한다는 걸 전혀 모릅니다. 쓰기 전에 주장을 검증함으로써 `write-for-ai` 는 문서를 단발성 산문 생성 작업이 아니라 *다음 세션과의 계약* 으로 다룹니다.

### 5. `setup-ci`

CI 는 이 플러그인 전체에서 가장 큰 생산성 이득입니다 — 제대로 세팅되면 *배포* 가 더 이상 생각할 거리가 아니라 `git push` 가 됩니다. 문제는 *제대로 세팅하는 것* 에서 대부분의 비개발자가 포기한다는 점입니다. Stack Overflow 에서 3 일 동안 러너가 뭔지 배우다가, concurrency 버그를 맞고, CI 는 너무 많다고 결론 내리고, 다시 서버에 SSH 해서 `git pull && sudo systemctl restart myapp` 을 수동으로 치는 생활로 돌아갑니다. `setup-ci` 는 그 3 일을 실제로 필요한 부분으로 압축합니다.

스킬은 개념 설명부터 시작합니다 — 평이한 말로, YAML 경험이 있다고 가정하지 않고. 러너가 무엇인지, Secrets 이 무엇인지, 왜 `workflow_run` 이 `push` 와 다른지, 왜 `concurrency` 그룹이 중요한지, 왜 모든 배포 뒤에 헬스 체크가 필요한지. 그다음 `package.json` / `requirements.txt` / `Cargo.toml` / `go.mod` 등에서 스택을 감지하고, 맞는 테스트와 린트 커맨드를 고르고, 동작하는 워크플로 두 개를 스캐폴드합니다. `test.yml` 은 모든 푸시와 PR 에서 테스트와 린트를 실행하며, 명시적 타임아웃과 읽기 좋은 잡 이름이 붙습니다. `deploy.yml` 은 성공 시 호스트에 맞는 패턴으로 배포합니다 — SSH, Vercel, Fly.io, Cloud Run, Netlify, 거기 있는 것 — concurrency 가드(두 배포가 경합하지 못하도록), SSH 키 정리 (크레덴셜이 잡 로그에 남지 않도록), 그리고 배포 후 헬스 체크 (사용자가 알기 전에 실패를 잡을 수 있도록) 를 함께 넣습니다.

`setup-ci` 가 하지 않는 두 가지가 있습니다. 둘 다 의도된 것입니다. Secret 을 대신 추가하지 않습니다 — Secret 은 GitHub UI 에 속한 것이고, 스킬은 어떤 Secret 을 추가해야 하고 뭐가 들어가야 하는지 말해 주지만, 크레덴셜 자체는 절대 직접 처리하지 않습니다. 그리고 호스트를 짐작하지 않습니다. 타겟이 명확하지 않으면 물어봅니다.

### 6. `manage-config-env`

새 프로젝트를 세팅하는 비개발자는 누구나 `main` 과 `dev` 중에서, `.env` 를 커밋할지 말지, 설정 파일과 환경변수 중에서, `feature/` 와 `feat/` 중에서, 그리고 첫 배포 전까지 아마 열두 번 더 *"두 가지 합리적 옵션"* 중에서 고르게 됩니다. 각 결정은 주의력에 붙는 세금이며, 사실 대부분은 결정이 필요 없습니다 — 95% 의 프로젝트가 같은 기본값으로 잘 돌아갑니다. `manage-config-env` 는 그 기본값을 미리 고르고, 각각을 한 문장으로 설명하고, 길을 비킵니다.

추천하는 기본값을 가능한 한 짧게 말씀드리면: 런타임에 절대 바뀌지 않는 상수는 코드에 들어갑니다. 로컬 전용 시크릿은 `.env` 에 들어가고 `.env.example` 은 커밋됩니다. 프로덕션 시크릿은 GitHub Secrets 에 들어갑니다. 환경마다 달라지는 런타임 설정은 환경변수로 들어갑니다. 브랜치는 GitHub Flow 를 따릅니다 (하나의 `main`, 짧은 `feature/...` / `fix/...` / `refactor/...`, 항상 PR 을 통한 머지). 그리고 `.gitignore` 는 흔한 것들 (`.env`, SSH 키, `__pycache__`, `node_modules`, `.venv`, `.DS_Store`, IDE 폴더) 로 미리 채워집니다. 이유를 물으시면 각 기본값에 한 줄씩 돌아옵니다 — 여러분 상황에 해당하는지 판단하기에 충분하되, 강의가 되기에는 부족한 분량으로.

새 프로젝트를 시작하시는 경우 스킬이 `.env.example`, `.gitignore`, 브랜치 전략 노트를 레포 루트에 스캐폴드할 수 있습니다. 기존 프로젝트에서 작업 중이시라면 이미 있는 것을 감사하고 기본값과의 불일치를 표시합니다 — 다만 여러분의 컨벤션을 조용히 다시 쓰지는 않습니다. 여러분은 이미 선택을 했고, 스킬은 그 선택이 자기 의견과 다를 때 그 사실만 말할 뿐입니다.

## 일상에서 쓰는 법

플러그인을 불러내는 방법은 세 가지입니다. 지금 원하시는 것이 얼마나 명확한지 순서대로 정리해 보겠습니다.

어디서부터 시작할지 모르실 때 — 새 레포, 지저분한 상태, 릴리스를 앞둔 긴장 — 는 `/vibesubin` 을 입력하십시오. 모든 것이 돌고 하나의 리포트가 돌아옵니다. 플러그인 이름이 붙은 모드입니다.

걸리는 게 있는데 말로 표현할 수 있을 때는 평범한 말로 말씀하시면 됩니다. *"정리해 주는데 아무것도 깨지지 않게 해 줘"* 는 `refactor-safely` 입니다. *"뭐 새는 거 있어?"* 는 `audit-security` 입니다. *"뭐부터 고칠까?"* 는 `fight-repo-rot` 입니다. *"README 써 줘"* 는 `write-for-ai` 입니다. *"배포 세팅해 줘"* 는 `setup-ci` 입니다. *"이 시크릿 어디 둬?"* 는 `manage-config-env` 입니다. 트리거 리스트를 외우실 필요는 없습니다. 스킬이 비개발자가 실제로 이런 요청을 꺼내는 방식에 반응하도록 만들어졌습니다.

쓰실 스킬을 이미 정확히 알고 계시다면 이름으로 바로 부르시면 됩니다: `/refactor-safely`, `/audit-security`, `/fight-repo-rot`, 이런 식으로. 스킬이 필요한 맥락 (*"어떤 파일입니까?"*, *"어느 호스트입니까?"*) 을 물은 뒤 작업에 들어갑니다.

### 스킬 출력의 생김새

플러그인의 모든 스킬은 같은 4 단계 출력 형태를 따릅니다: 무엇을 했는지, 무엇을 찾았는지, 무엇을 검증했는지, 다음에 무엇을 하면 좋을지. 그게 전부입니다. 만약 이 네 단계 없이 산문 덩어리만 나오면 그건 버그입니다 — [이슈](https://github.com/subinium/vibesubin/issues) 를 열어 주시면 고쳐집니다.

## 자주 등장하는 워크플로

스킬들은 서로에게 자연스럽게 핸드오프합니다. 같은 체인이 반복해서 등장합니다.

**레포 정리하기.** `fight-repo-rot` 으로 가장 심한 오프닝을 찾고, `refactor-safely` 가 4 단계 검증과 함께 그걸 고치도록 두시고, 마지막에 `write-for-ai` 로 커밋과 PR 설명을 쓰시면 됩니다.

**릴리스 준비.** `audit-security` 로 시크릿과 흔한 구멍을 스캔하고, 크리티컬한 것은 `refactor-safely` 가 고치게 하고, `setup-ci` 가 다음 번에는 회귀를 잡아낼 워크플로가 있는지 확인하게 하십시오.

**방금 넘겨받은 레포에 적응하기.** `/vibesubin` 으로 전체 스윕을 돌리시고, 그 발견 사항을 기반으로 `write-for-ai` 가 없는 `CLAUDE.md` 를 채우게 하시고, 그다음 `manage-config-env` 로 `.gitignore`, `.env.example`, 브랜치 전략을 감사하게 하십시오.

**새 프로젝트를 처음부터 시작하기.** `manage-config-env` 가 `.env.example`, `.gitignore`, 브랜치 레이아웃을 스캐폴드합니다. `setup-ci` 가 테스트와 배포 워크플로를 깔아줍니다. `write-for-ai` 가 README 와 `CLAUDE.md` 를 써서 다음 세션 — 사람이든 AI 든 — 이 프로젝트를 역추적할 필요가 없게 해 줍니다.

이 체인을 직접 계획하실 필요는 없습니다. 스킬들이 적절한 시점에 다음 핸드오프를 제안하므로, 대개는 *"예"* 라고만 하시면 됩니다.

## 직접 스킬 추가하기

플러그인은 자라날 수 있도록 만들어져 있습니다. 새 스킬은 `plugins/vibesubin/skills/<skill-name>/` 아래의 자체 완결 디렉토리입니다. 거기에 두고 에이전트를 재시작하시면 `/vibesubin` 과 자동완성 메뉴 모두 픽업합니다 — 건드려야 할 레지스트리 파일은 없습니다.

최소한의 구조는 이렇습니다:

```
plugins/vibesubin/skills/<skill-name>/
├── SKILL.md              # 필수, 500 줄 아래, YAML frontmatter 포함
├── references/           # 선택, SKILL.md 가 링크할 때만 로드되는 상세 문서
├── scripts/              # 선택, 실행되는 헬퍼 스크립트
└── templates/            # 선택, 사용자 프로젝트로 복사되는 파일
```

최소한의 `SKILL.md` 는 이렇습니다:

```markdown
---
name: my-new-skill
description: 한 문장으로 이 스킬이 언제 트리거되고 무엇을 하는지. 트리거 문구를 끝에 포함시켜 AI 가 사용자 요청을 스킬에 매칭할 수 있도록 합니다.
---

# my-new-skill

## 언제 실행되는가
...

## 무엇을 하는가
1. ...
2. ...

## 하지 않는 것
- ...

## 출력 형태
1. 무엇을 했는지
2. 무엇을 찾았는지
3. 무엇을 검증했는지
4. 다음에 무엇을 해야 하는지
```

새 스킬이 물려받아야 할 규칙 몇 가지 — 코드로 강제되는 것이 아니라 플러그인이 일관성을 유지하는 방식입니다. *완료* 는 주장이 아니라 검증입니다. 변경을 만드는 모든 스킬은 증거를 만들어야 합니다. 최상위 `SKILL.md` 는 500 줄 아래로 유지되고, 깊이는 `references/` 로 내려갑니다. 출력은 네 단계 형태를 따릅니다 — 무엇을 했는지 / 무엇을 찾았는지 / 무엇을 검증했는지 / 무엇을 해야 하는지. `/vibesubin` 에서 호출될 수 있는 스킬은 반드시 읽기 전용 모드가 있어야 합니다. 그리고 모든 스킬이 무엇을 확인했고 무엇을 확인하지 않았으며 어디까지 자신할 수 있는지에 대해 명시적이어야 합니다.

새 스킬을 메인 플러그인에 포함시키고 싶으시면, 트리거 문구, 하는 일, 실제 사용 예시를 함께 적어 이슈를 열어 주십시오. 아래 [기여](#기여) 를 참고하시면 됩니다.

## 트러블슈팅

**스킬이 자동완성에 보이지 않음.** 모든 에이전트 세션을 닫고 몇 초 뒤에 새 세션을 여십시오. 스킬은 세션 단위로 캐싱됩니다.

**`install.sh` 가 "Permission denied" 라고 함.** `chmod +x install.sh` 를 실행하고 `bash install.sh` 를 재시도하십시오.

**마켓플레이스가 플러그인이 "already exists" 라고 함.** 이전에 등록하신 것입니다. `/plugin marketplace update` 로 넘어가시면 됩니다.

**`npx skills add` 가 스킬을 못 찾는다고 함.** 사용 중인 에이전트가 아직 `skills.sh` 지원 목록에 없을 수 있습니다. `npx skills --help` 가 현재 타겟을 보여줍니다.

**두 경로로 중복 설치했는데 뭔가 깨짐.** 수동 클론에서 `bash uninstall.sh --to all` 을 돌려 심볼릭 링크를 정리하시고, 하나의 경로만 골라 유지하십시오.

**`install.sh --to codex` 이후에도 Codex CLI 가 스킬을 못 봄.** 실행 후 `~/.codex/skills/` 가 존재하는지 확인하십시오. 여러분의 Codex 버전이 다른 skills 디렉토리를 쓴다면 `install.sh` 의 타겟을 패치하시거나 클론을 직접 symlink 하시고, 버전과 실제 경로를 적어 이슈를 열어 주시면 스크립트가 고쳐집니다.

**스킬이 사실과 다른 것을 보고함.** 블로커급 버그입니다. 실행한 커맨드, 받은 출력, 기대했던 것을 적어 이슈를 열어 주십시오.

그 외의 문제는 정확한 오류와 시도한 내용을 적어 [이슈](https://github.com/subinium/vibesubin/issues) 를 열어 주십시오.

## 내부 구조

플러그인은 [`plugins/vibesubin/`](./plugins/vibesubin) 에 살고, [`.claude-plugin/marketplace.json`](./.claude-plugin/marketplace.json) 매니페스트를 통해 배포됩니다 — Claude Code 마켓플레이스가 이 레포에 무엇이 있는지 알기 위해 읽는 파일입니다. 각 스킬은 독립된 디렉토리입니다:

```
skills/<skill-name>/
├── SKILL.md              # 필수, 스킬이 트리거되면 AI 가 읽는 진입점
├── references/           # 선택, 필요할 때만 로드되는 상세 문서
├── scripts/              # 선택, AI 가 실행하는 헬퍼 스크립트
└── templates/            # 선택, 사용자 프로젝트로 복사되는 산출물
```

플러그인이 일관되게 따르는 몇 가지 컨벤션을 말씀드리겠습니다. `SKILL.md` 는 500 줄 아래로 유지되어 AI 가 빠르게 로드할 수 있게 합니다. 더 깊은 내용은 `references/` 로 내려갑니다. 참조 파일이 또 다른 참조 파일로 체이닝되지 않습니다 — 한 단계 깊이, 재귀적 토끼굴 없음. 긴 체인은 부분만 읽히는 경향이 있기 때문입니다. 스크립트는 *실행 대상* 이지 *읽기 대상* 이 아닙니다. 스킬이 *"이 스크립트를 실행하십시오"* 라고 할 때 로직이 지시 안에 복사되지 않습니다. 템플릿은 산출물입니다. `templates/` 아래 파일들은 플레이스홀더가 채워진 뒤 여러분 프로젝트로 복사되며, 산문으로 설명되지 않습니다. 그리고 최상위 지시는 언어 중립을 유지하고, 언어별 세부는 `references/language-*.md` 로 내려갑니다.

## 이 플러그인이 다루지 않는 것

vibesubin 은 의도적으로 범위가 정해져 있고, 다루지 않는 것에 대해 솔직합니다. 규제 준수 감사 — SOC 2, HIPAA, PCI-DSS — 가 필요하시면 제대로 된 컴플라이언스 도구를 쓰십시오. 모의 침투 테스트가 필요하시면 전문 펜테스터를 고용하십시오. 풀 디자인 시스템이나 UI 컴포넌트 라이브러리는 범위 밖입니다. 언어 특화된 깊은 리팩토링 (Java 의 OpenRewrite, TypeScript 의 AST codemod) 은 해당 언어 도구에 속합니다. vibesubin 은 언어 독립적으로 설계되었습니다. 프로젝트 관리나 이슈 트래킹도 범위 밖입니다.

범위 밖 요청을 하시면 플러그인이 솔직하게 말합니다. 그리고 이 플러그인이 *다뤄야 한다고* 생각하시는 것이 있다면 이슈를 여십시오 — 그것이 새 스킬이 추가되는 방식입니다.

## 철학

플러그인의 모든 스킬이 기반하는 규칙 몇 가지입니다. 외우실 필요는 없습니다 — 현재 스킬들이 일관되게 행동하고, 나중에 추가되는 스킬도 같은 형태를 물려받도록 하기 위한 기준입니다.

AI 는 선의의 주니어 개발자입니다. 주니어는 열심히 일합니다. 동시에, 멈춰서 물어야 할 때 자기 판단으로 진행하기도 합니다. 이 플러그인은 조금 더 신중한 손이라면 멈춰서 물을 만한 순간을 명시적으로 넣었습니다.

*완료* 는 주장이 아니라 증명입니다. 작업이 "완료" 라고 말하려면 실행 결과가 뒷받침되어야 합니다 — 통과한 테스트, 일치하는 해시, 살아 있는 `200 OK`. 이 플러그인의 관점에서 증거 없는 단언은 버그입니다.

문서는 사람만이 아니라 다음 AI 세션을 위한 것입니다. 현재 대화의 맥락은 세션이 끝나면 증발합니다. README, 커밋, PR 본문은 새 세션이 처음부터 맥락을 재구성할 수 있도록 작성됩니다.

기존 컨벤션이 기본값입니다. 기여자는 (푸시 권한을 받은 바이브 코더를 포함해서) 관리자의 브랜치 전략이나 파일 구조를 임의로 다시 쓰지 않습니다.

경로는 상대, 값은 환경변수, 버전은 고정입니다. 하드코딩된 절대 경로, IP 리터럴, 고정되지 않은 의존성은 시한폭탄이며 플러그인은 그렇게 다룹니다.

그리고 리팩토링은 재귀적 증명입니다. 옮겨진 모든 심볼은 symbol-set diff, AST diff, smoke test, call-site closure 네 가지를 전부 통과해야 합니다. 하나라도 실패하면 리팩토링은 미완입니다.

플러그인은 [적극적으로 유지보수됩니다](./MAINTENANCE.md). 리팩토링 도구도, 스킬 시스템도, LLM 실패 양상도 계속 바뀝니다. 각 스킬은 정해진 주기로 리뷰됩니다.

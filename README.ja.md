# vibesubin

AI アシスタントに、リファクタリング・監査・デプロイを「本来こうあってほしい」やり方で仕込むための、ポータブルなスキルプラグインです。しかも `/vibesubin` ひとつで全部まとめて走らせられます。

開発者としての訓練は受けていないけれど、AI を使って実際にものを出荷している人のために作りました。自分が AI とコーディングするときに実践している習慣をスキルとしてパッケージしたものです。ルールをいちいち覚えなくても、アシスタントがそのまま従ってくれます。

同じ `SKILL.md` が **Claude Code**、**Codex CLI**、そして **[skills.sh](https://skills.sh) がサポートするあらゆるエージェント**(Cursor、Copilot、Cline など)で動きます。一度書けば、どのホストでも拾われます。

## クイックスタート

[Claude Code](https://code.claude.com) をインストールして、次を実行:

```
/plugin marketplace add subinium/vibesubin
/plugin install vibesubin@vibesubin
```

リポジトリを開いて `/vibesubin` と打つだけ。全スキルが並列・読み取り専用でコードを走査し、優先順位付きのレポートひとつにまとめて返ってきます。リストから承認するまで、コードには一切触れません。スキルに実際に *作業* をさせたいときは、名前で直接呼び出してください(`/refactor-verify`、`/setup-ci` など)。そうすれば、ファイルを直接編集します。

Codex CLI、Cursor、Copilot、Cline を使っている場合は [Install](#install) セクションへどうぞ。

## vibesubin とは、そうでないもの

AI スキルの小さなバンドルです。`SKILL.md` ファイル群で構成されていて、エージェントがリクエストにマッチしたときに自動で拾います。トリガーフレーズを暗記する必要はありません。普段の言葉で *「このファイル安全に分割して」*、*「なんか漏れてない?」*、*「デプロイ組んで」* と言えば、適切なスキルが走ります。

全スキル共通のルール: **証拠を見せられるまで「完了」とは言いません。** リファクタは AI がファイルを書き換えたから終わるのではなく、4 つの独立したチェックで「何も落ちていない・ずれていない・誤配線していない」が確認できて初めて終わります。セキュリティスイープは「vibe」で書いた段落ではなく、各ヒットが「本物」「誤検知」「要人間確認」のどれかに仕分けされ、ファイル名と行番号がついたリストです。

**そうでないもの**: SaaS ではありません(何もマシンから出ません)。コンプライアンスツールでもありません(SOC 2 / HIPAA は対象外)。コードジェネレータでもありません。すでにあるリポジトリを改善するためのものです。

### 読み取り専用のスイープと、実際にファイルを編集するスキル

早めにはっきりさせておきたいポイントです。プラグインの使い方は 2 通りあり、挙動がまったく違います。

- **スイープモード(`/vibesubin`)**。全スキルが並列で走りますが、すべて *読み取り専用* です。返ってくるのは findings であって、修正ではありません。レポートから項目を承認するまで、リポジトリには何の変更も入りません。「正直なセカンドオピニオンが欲しい」モードです。
- **直接呼び出し(`/refactor-verify`、`/setup-ci`、`/write-for-ai`、`/manage-secrets-env`、`/project-conventions`、`/unify-design`)**。スキルが本来の仕事をフルで行います。これにはファイル編集も含まれます。`refactor-verify` は依存ツリー全体にわたってコードを書き換えます。`setup-ci` は `.github/workflows/` に動く YAML を配置します。`write-for-ai` は README を編集します。`manage-secrets-env` は `.env.example`、`.gitignore` を足場組みし、シークレットのライフサイクル全体を回します。`project-conventions` は Dependabot を足場組みし、依存のピン留めを強制し、ハードコードされたパスを直します。`unify-design` はトークンファイルを足場組みし、コンポーネントをトークン参照に書き換えます。こちらは「実際に作業する」モードです。
- **直接呼び出し、ホスト固有(`/codex-fix`)**。一つのワークフローのための薄い wrapper: 現在のブランチに `/codex:rescue` を実行して findings を受け取り、`refactor-verify` の review-driven fix mode にハンドオフして 4-check 検証とともに解決します。**Claude Code と Codex プラグインがインストールされた環境でのみ動作**し、他のあらゆるホストでは一行の fallback を出してきれいに抜けます。ポータブルエンジンは `/refactor-verify` 側にあるので、Codex 以外のレビューソース(PR レビュー、Sentry、スキャナー、手で貼り付けたメモ)で作業する場合は `/refactor-verify` を直接呼び出してください。

どちらの呼び方をしても絶対に編集しないスキルが 3 つあります: **`fight-repo-rot`**(純粋な診断専用 — デッドコードと匂いを見つけて、削除は `refactor-verify` にハンドオフ)、**`audit-security`**(静的な仕分けレポートのみ)、**`manage-assets`**(肥大化レポートのみ — 履歴の書き換えもファイル削除もしません)。6 つのスキル — `refactor-verify`、`setup-ci`、`write-for-ai`、`manage-secrets-env`、`project-conventions`、`unify-design` — は、直接呼べば本物のワーカースキルとしてファイルを編集し、スイープ経由なら読み取り専用のレポーターとして動きます。1 つのスキル — `codex-fix` — は Claude Code と Codex プラグイン専用の薄い wrapper で、直接呼び出しのときだけ動き、スイープにはまったく参加せず、他のホストでは一行の fallback を出して抜けます。

新しいスキルは `plugins/vibesubin/skills/` に置けば `/vibesubin` が自動で拾います。

## `/vibesubin` コマンド

プラグイン内のすべてのスキルをリポジトリに対して並列実行し、発見事項をひとつのレポートにマージし、承認を待つ。そういう一語です。

動きはこう。スキルはまず自分が何をしようとしているかを一文で伝え、起動前にスコープを絞る余地を与えます(*「src/api だけ」*、*「security と repo rot だけ」* とか)。指定がなければ、すべての専門スキルを独立したタスクエージェントとして展開します。各実行は分離されていて、エージェント間の汚染はありません。このパスでは全スキルが読み取り専用で、見つけるだけで直しません。動かないスキル — 例えばテストスイートが起動すらしないケース — はそれ自体を findings として報告して静かに退きます。壊れた 1 個がスイープ全体を止めることはありません。

帰ってきた findings は synthesis ルールに従ってマージされます。Critical はカテゴリに関係なく最上位に浮きます(漏れたシークレット > ホットスポット > docstring 欠如)。複数の専門スキルが同じファイルを独立に指摘したら、そのファイルの優先度は跳ね上がります。findings はファイル単位でグルーピング — 人間が直すのはファイルであってカテゴリではないからです。各項目には具体的な推奨アクションがひとつ付きます。

返ってくるのは markdown のレポートひとつ: vibe check の段落、専門スキルごとの信号機一行、トップ 10 修正リスト、推奨順序、そして一文の総評。問題がなければそう書いてあります。承認するまでリポジトリには何も起きません。

レポートの完全なテンプレートと synthesis ルールは [umbrella の `SKILL.md`](./plugins/vibesubin/skills/vibesubin/SKILL.md) にあります。

**スイープか単体か**。オープンエンドな質問にはスイープ: *「このリポジトリ引き継いだばかり」*、*「出荷していい?」*、*「セカンドオピニオンが欲しい」*。やりたいことが決まっているなら単体で呼ぶ: *「このファイルリファクタして」* → `/refactor-verify`。*「`.env` を push しちゃった」* → 緊急度重視で `/audit-security`。*「README 書いて」* → `/write-for-ai`。

**ハーシュモード**。スイープはデフォルトではバランスの取れたトーンで走ります — 正直だけど、あたたかい。ハーシュモードを頼むと、言葉を和らげるのをやめます: *「`/vibesubin harsh`」*、*「容赦なくレビューして」*、*「オブラートに包まないで」*、*「매운 맛으로」*、*「厳しめで」*。レポートは引き続き読み取り専用で、根拠も同じ証拠に裏付けられていますが、ぼかした言い回しを落とし、一番ひどい findings から書き始め、本物の問題が残っているのに *「looks fine」* で締めることを拒みます。オプトイン専用で、勝手にハーシュになることはありません。

## Install

| パス | 対象エージェント | 向いている場面 |
|---|---|---|
| **A. Claude Code marketplace** | Claude Code | 最もシンプル、自動更新 |
| **B. `skills.sh`** | Cursor、Copilot、Cline、Codex CLI、Claude Code | エージェント横断 |
| **C. 手動シンボリックリンク** | Claude Code / Codex CLI | プラグイン自体をいじるとき |

**A — Claude Code marketplace**(推奨)

```
/plugin marketplace add subinium/vibesubin
/plugin install vibesubin@vibesubin
```

更新は `/plugin marketplace update`、アンインストールは `/plugin uninstall vibesubin`。

**B — skills.sh**(エージェント横断、[skills.sh](https://skills.sh) 経由)

```bash
npx skills add subinium/vibesubin
```

同じコマンドを再実行すると更新されます。削除は `npx skills remove vibesubin`。対応ホストは `npx skills --help` で確認できます。

**C — 手動シンボリックリンク**(編集可能、オフライン、`git pull` でそのまま反映)

```bash
git clone https://github.com/subinium/vibesubin.git
cd vibesubin
bash install.sh                  # Claude Code(デフォルト)
bash install.sh --to codex       # Codex CLI
bash install.sh --to all         # 両方
bash install.sh --dry-run        # プレビュー
```

更新は `git pull`。アンインストールは `bash uninstall.sh [--to codex|all]` — スクリプトが作ったシンボリックリンクだけが削除されます。

インストールでつまずいたら、まずはエージェントのセッションを全部閉じて、新しいセッションを開き直してください(スキルはセッション単位でキャッシュされます)。それでもダメなら [issue を立ててください](https://github.com/subinium/vibesubin/issues)。

## スキル一覧

各スキルは [`plugins/vibesubin/skills/`](./plugins/vibesubin/skills/) にあります。以下はユーザー向けの説明で、完全な方法論は各スキルの `SKILL.md`、詳細な参考資料は同じ階層の `references/` フォルダにあります。スキルファイルを読む必要はありません — AI が読むからです — ただオープンソースなので気になれば覗いてみてください。

### 1. `refactor-verify`

一番強く擁護したいスキルです。AI がコードに触れるときの最大の失敗モードは、静かな失敗です。関数が移動されて、リネームも通って、テストも通って、3 週間後に誰かが古い名前を指したままのコードパスを踏んで全部落ちる。`refactor-verify` は、このタイプの失敗を起こしようがなくするために存在します。

2 種類の変更をカバーします — 構造的リファクタリング(move / rename / split / merge / extract / inline)と、安全な削除(`fight-repo-rot` が dead と確認したコードの削除)。両方とも同じ 4 チェックの証明を使います。

何かに触れる前に、現状のスナップショットを取ります。どんな関数が存在するか、どのファイルにいるか、今この瞬間テストは通っているか、lint は何と言っているか。これが *before* の絵です。続いて変更を依存ツリーに分解し、葉から上に向かって進めるので、実行の途中で中途半端な状態になることがありません。

1 ステップごとに、4 つの独立したチェックが走ります。ひとつ目はシンボル集合を走査して、以前存在したすべてのパブリック名が今も存在する(または意図的に削除された)ことを確認。ふたつ目は移動 / 維持されたコードが空白を除きビット単位で同一であることを確認。みっつ目は typecheck、lint、テスト、スモークランを再実行。そして 4 つ目 — 実際のバグを一番捕まえるやつ — は、影響を受けたすべてのシンボルの呼び出し側を歩いて、正しい場所を指しているか(削除されていれば何も指していないか)を確認します。4 つのうち 1 つでも落ちたら、スキルは先に進みません。

やらないこと: 頼んでいないファイルに触る、どれか 1 つでも落ちているのに「完了」と言う、テストスイートが起動しないのに結果をでっち上げる、移動中に関数の中身を書き換える、`fight-repo-rot` が LOW 信頼度でフラグしたコードを人間のレビューなしに削除する。

### 2. `audit-security`

エンタープライズのセキュリティスキャナにはノイズ問題があります。500 件フラグを立てて、うち 490 件が誤検知なので、1 週間もすれば誰も見なくなり、本物の critical が溺れます。`audit-security` はその逆の形です。人間が実際にやらかすミスを捕まえる、手作業で選ばれた短いパターン集で、各ヒットを「本物」「誤検知」「要人間確認」に仕分けし、一行の理由を添えます。

チェック対象は、明らかなもの(コミットに入ったシークレット、文字列連結で組まれた SQL、信頼できない入力に対する `eval` / `exec` / `pickle.loads`)、そこまで明らかでないもの(ユーザー入力で制御されるファイルパス、HTML に未エスケープで埋め込まれる入力、`httpOnly` / `Secure` のない Cookie、ワイルドカード CORS)、そして忘れがちなもの(git 履歴の中の `.env`、`.pem` や SSH 鍵)。重大度は平易な言葉で書きます — *「見知らぬ他人が全ユーザーのデータを読める」* が *「CWE-89」* より伝わるからです。

ペネトレーションテストの代わりにはなりません。静的なスイープで、ネットワークもランタイムも触りません。そして、都合の悪いものを *「looks fine」* の総評の後ろに隠したりもしません。

### 3. `fight-repo-rot`

一にデッドコード検出、二に散らかり検出、そして常に純粋な診断です。誰も呼んでいない関数、誰も import していないファイル、孤立したモジュール、消費者のいない export、manifest にあるのに一度も import されていない依存 — こういうものを見つけるために作られています。その上で、よくある匂いもフラグします: 神ファイル、神関数、ハードコードされた絶対パス、リテラル IP、6 ヶ月以上古い `TODO` / `FIXME` / `HACK`、そして *「次にやらかす可能性が一番高い」* リストの上位ファイル(変更頻度が高くて、かつ複雑なやつら)、さらにテストの腐敗(デッドテスト、古いフィクスチャ、孤立したスナップショット)。

デッドコード候補には必ず信頼度タグが付きます:

- **HIGH** — どこからも参照されていない(grep、LSP、import グラフ、テスト、config ファイルのすべてで)。`refactor-verify` に削除を渡して安全です。
- **MEDIUM** — テストからしか参照されていない、または言語が動的ディスパッチを使う(Python、Ruby、ゆるい JS)。削除前にオペレータの確認が必要。
- **LOW** — export されたシンボル、生成コード、リフレクション / DI / アノテーションが絡む。人間のレビュー必須。自動ハンドオフは絶対にしません。

このスキルは意図的に手を出しません: 編集しない、削除しない、検証も走らせない。問題を証拠付きで浮かび上がらせて、削除と分割は `refactor-verify` に、ハードコードされたパスの修正は `project-conventions` に、CVE の依存腐敗は `audit-security` にハンドオフします。リストを承認するまで、何にも触れません。

### 4. `write-for-ai`

ほとんどのドキュメントは、一度だけ流し読みする人間の読者向けに書かれています。AI の読み方は違います。セッションごとに毎回新鮮にパースし直し、散文よりテーブルを好み、バッククォートで囲まれた明示的なファイルパスを追い、*「never do X」* がストーリーの中に埋もれているより宣言的に書かれていることを必要とします。`write-for-ai` はその読者に向けて書きます — そして、構造は構造なので、結果的に人間にとっても読みやすくなります。

書いてほしいものを渡すと — README、commit、PR description、architecture doc、`CLAUDE.md`、`AGENTS.md` — まず新鮮な AI セッションのつもりでリポジトリを冷たい状態から読みます。それから不変条件を抽出します。プロジェクトが *何を* するかだけでなく、*どんなルールに従っているか* も。関連するテンプレートを埋め、そして何より大事なこととして、書く前に各主張を検証します。README に `pnpm test` でテストが走ると書くなら、スキルは先に `pnpm test` を実行します。

そしてもうひとつ — このバージョンから入った原則: **誇張しない、根拠なしに売り込まない**。*「fast」*、*「robust」*、*「production-ready」*、*「best-in-class」* のようなマーケティング語は、ベンチマークか実例が添えられているときだけ使い、そうでなければ削除します。能力主張には検証コマンドが並びます。曖昧な数字(「数百ユーザー」)は具体的な数字に置き換えるか、消します。

これが防ぐもの: AI に README を書き直させたら見事な仕事をしてくれたけど、環境変数についての段落がこっそり消えていた。何が書いてあったか覚えていないから気づかない。次の AI セッションは新しい README から始まって、env のレイアウトがあることすら知らない — そういう事故です。

### 5. `setup-ci`

CI はこのプラグインの中で最大の生産性向上です。正しく組めば、*deploy* は考えるものではなくなって、`git push` そのものになります。残念ながら、*正しく組む* のところで、ほとんどの非開発者は諦めます。

このスキルはまず概念を平易な言葉で説明します — runner とは何か、Secrets とは何か、なぜ `concurrency` グループが重要か、なぜ毎デプロイ後にヘルスチェックをかけたいか。それから `package.json` / `requirements.txt` / `Cargo.toml` / `go.mod` からスタックを検出し、適切なテスト / lint コマンドを選び、動くワークフローを 2 本足場組みします。`test.yml` は push と PR ごとにテストと lint を明示的なタイムアウト付きで実行。`deploy.yml` は成功時にホストに合ったパターンでデプロイ — SSH、Vercel、Fly.io、Cloud Run、Netlify — concurrency ガード、SSH 鍵のクリーンアップ、デプロイ後のヘルスチェック付きで。

意図的にやらないことが 2 つ: Secrets を代わりに登録しません(Secrets は GitHub UI に置くものです。スキルはどの Secrets が必要で何を入れるか教えますが、認証情報自体には触れません)。そして、ホストを推測しません。

### 6. `manage-secrets-env`

シークレットは「この値どこに置くべき?」という問いの中で、一番賭け金が高い部分です — 置く場所を間違えるのはスタイルの好みの問題ではなく、インシデントです。`manage-secrets-env` がその部分を専任します: 4 バケツの判定ツリー(ソース定数 / 環境変数 / ローカル `.env` / CI シークレットストア)、`.env.example` ↔ `.env` のドリフトチェック、デフォルトで安全な `.gitignore` テンプレート、そしてシークレットのライフサイクル全体(追加 / 更新 / ローテーション / 削除 / バケツ間移行 / 環境横断のドリフト監査 / 新環境のプロビジョニング)。

デフォルトの最短版: ランタイムで変わらない定数はコードの中。ローカルだけのシークレットは `.env` にコミット済み `.env.example` と一緒に。本番のシークレットは CI プロバイダのシークレットストアに。環境ごとに変わるランタイム値は環境変数として。`.gitignore` はシークレット形のエントリが事前に全部埋まった状態で出発します。

新規プロジェクトでは `.env.example`、`.gitignore`、起動時バリデーションを足場組みします。既存プロジェクトでは現状を監査し、すでに tracked になっているシークレットファイルはインシデント級の findings としてフラグし、漏洩が進行中なら `audit-security` にハンドオフします。

### 7. `project-conventions`

`manage-secrets-env` のロースクテス側の相棒です。プロジェクトにはシークレットとは関係ない構造的な決定がいくつもあります: main と dev のどっちで作業するか、依存をピン留めするかフロートさせるか、ドメイン単位のディレクトリレイアウトかタイプ単位か、ソースに絶対パスが紛れ込むのをどう止めるか。大半には 95% のプロジェクトで通用する答えが 1 つあり、それを選ぶことにセッションを 1 つ使う必要はありません。

デフォルト: GitHub Flow(`main` + 寿命の短い feature ブランチ、`dev` は無し)、本番依存は厳密にピン留めし lockfile をコミット、Dependabot か Renovate を月次で、ドメインファーストのディレクトリレイアウト、ソースに絶対パスを書かない。各ルールには一文の理由が付きます。

新規プロジェクトでは `dependabot.yml` とブランチ戦略のメモを足場組みします。既存プロジェクトではブランチの逸脱、ピン留めされていない依存、ディレクトリの匂い、ハードコードされたパスを監査し、複数ファイルにまたがる修正は `refactor-verify` に渡して、verification パスなしでは何も書き換わらないようにします。

### 8. `manage-assets`

肥大化検出器であって、コード分析器ではありません。リポジトリを遅くしているのはコードではなくバイナリです — 去年誰かが入れた 300 MB の SQLite、`.gitignore` をすり抜けた `dist/` ディレクトリ、本当は LFS に行くべきだった tracked `.psd`。`manage-assets` はその重さを浮かび上がらせます: ワーキングツリー内のファイルサイズ、git 履歴内の blob サイズ(見えない方)、LFS 移行候補、アセットディレクトリの増大、重複バイナリ。

このスキルは **診断専用** です。ファイルを削除しない、履歴を書き換えない、`git filter-repo` や `git lfs migrate` を走らせない。オペレータが削除を承認したとき、破壊的な作業は `refactor-verify`(履歴の書き換えのような destructive 操作の verification を担う)、`.gitignore` 絡みの発見は `manage-secrets-env`、参照されていないアセットは `fight-repo-rot` にハンドオフされます。

オープンソース公開の直前に特に相性がいい — 遅い接続での最初の `git clone` は、リポジトリがどれだけ重たくなったかを正直に測ってくれます。

### 9. `unify-design`

あらゆる vibe-coding プロジェクトが最終的に必要とし、毎回後回しにされるもののための Web スキル: デザインシステムは 1 つ、参照は一貫、ドリフトなし。ほとんどのプロジェクトは、primary blue が 3 つ、Button 実装が 2 つ、微妙に違う padding が 5 種類、ロゴが 6 ファイルに生の `<img>` として貼られていて、2 ページのナビゲーションが一致しない、そんな状態で出荷されています。個別に見れば目立たず、初見の訪問者にはすぐバレます。

`unify-design` はプロジェクトの BI(ブランドアイデンティティ)— 色、間隔、タイポグラフィ、radius、shadow、breakpoint、そして全ページに出てくる中心コンポーネント — を単一の source of truth として扱い、ドリフトをトークン参照に書き戻します。フレームワークは自動で検出します(Tailwind v3 と v4、CSS Modules、styled-components、Emotion、Material UI、Chakra UI、カスタムプロパティを使った vanilla CSS)。プロジェクトのイディオムをそのまま使い、外のパターンを持ち込みません。

3 つのことをします。まず、source of truth を確立する — トークンファイルがなければ、オピニオネイテッドなデフォルト(spacing スケール、typography スケール、radius スケール)と一緒に足場組みし、推測できない 2 つの値 — primary color と display font — だけをオペレータに尋ねます。次にドリフトを監査する — トークンファイルの外にあるハードコードされた hex、`w-[432px]` のような Tailwind arbitrary value、インラインスタイルオブジェクト、重複した Button / Card / Nav / Logo コンポーネント、明らかにコピペミスで生まれた near-match カラー。そして 3 つ目、ドリフトを直す — 小さな置換は直接適用し、複数ファイルにまたがるリファクタリングは `refactor-verify` に渡して、トークンのリネームやコンポーネント統合が call-site 検証を通るようにします。

意図的にやらない 2 つ: プロジェクトに BI そのものが無い場合、ブランドを発明しない(尋ねます)。そしてフレームワークを乗り換えさせない — styled-components のプロジェクトなら theme オブジェクトを使い、Tailwind に引っ越させたりしません。

### 10. `codex-fix`

パックで最初にして意図的に唯一のホスト固有スキル。一人のメンテナーの一つのワークフローが、一つのワーカースロットを使うに値するほど頻繁だから存在します: *「編集のひとまとまりを終えた。Codex でセカンドモデルのレビューを走らせる。findings を返してもらう。Claude に検証付きで解決させる。」* Codex の出力と Claude の解決の間のコピーペーストがセッションごとに摩擦になっていたので、`codex-fix` がそのステップを自動化します。

意図的に薄いのです — 100 行ほど — 実質的な作業はすべて `refactor-verify` の review-driven fix mode に委譲しているからです。この wrapper が所有するのは Codex 固有の glue だけ: ホストチェック、現在のブランチ diff にスコープした `/codex:rescue` のテンプレートプロンプト、出力の収集、ハンドオフ。findings のパース、real / false-positive / defer / duplicate のトリアージ、`git blame` で各項目をコミットにマッピング、依存ツリーの計画、葉から上に実行しながら 4-check 検証、レビュー項目への back-reference を付けてコミット — これらは全部 `refactor-verify` の仕事です。

**ホスト要件**。このスキルは **Claude Code と Codex プラグインがインストールされた環境** でのみ起動します。他のあらゆるホスト(Codex CLI 自体、Cursor、Copilot、Cline、または Codex プラグインの無い Claude Code)では、最初のアクションが一行の fallback と exit です: *「Codex プラグインが検出されませんでした — このスキルは Claude Code と Codex 専用です。別のソースのレビューを解決するには、findings を `/refactor-verify` に直接渡してください。」* ハングしませんし、大声でエラーを出すこともありません。

**`/codex-fix` と `/refactor-verify` を直接呼び出すのと、どちらを選ぶか**。Claude Code と Codex の環境にいて、レビューソースが Codex である場合は `/codex-fix`。レビューソースが他のもの(人間の PR レビュー、Sentry アラート、`gitleaks`、Semgrep、GitHub Advanced Security、手書きのメモ)である場合は `/refactor-verify` を findings を貼り付けて直接呼び出してください。エンジンは同じで、入力アダプタだけが違います。

これはパックで最初の **host-locked** スキルで、[`docs/PHILOSOPHY.md`](./docs/PHILOSOPHY.md) のルール 9 — *「ポータブルなエンジンはホスト固有のラッパーを持てる」* — のパターンでカバーされています。他の入力ソース(Sentry、gitleaks など)のための将来のラッパーも同じ形にすべきです: ポータブルエンジンの周りを囲む薄い glue、フロントマターでホスト依存性を明示的に宣言、最初のアクションは graceful fallback。

## 日常の使い方

呼び出し方は 3 通り、知っている情報量の少ない順に。

どこから始めていいか分からない? `/vibesubin` と打てば全部走ります。何が気になっているか分かっている? 普段の言葉で言ってください — *「ここ整理して、ただし壊さないで」* で `refactor-verify`、*「シークレット漏れてない?」* で `audit-security`、*「何から直すべき?」* で `fight-repo-rot` が起動します。使いたいスキルが決まっている? 名前で呼んでください: `/refactor-verify`、`/audit-security` など。

全スキルは 4 部構成の出力に従います: 何をしたか、何を見つけたか、何を検証したか、次に何をすべきか。もしスキルがこの 4 つを含まない散文の壁を返してきたら、それはバグです — [issue を立ててください](https://github.com/subinium/vibesubin/issues)。

### よくあるワークフロー

- **リポジトリの掃除**。`fight-repo-rot` が最悪の箇所を見つける → `refactor-verify` が検証付きで直す → `write-for-ai` がコミットと PR を書く。
- **リリース準備**。`audit-security` でシークレット確認 → critical なものは `refactor-verify` で対応 → `setup-ci` で次の退化を捕まえる体制に。
- **新しいリポジトリへのオンボーディング**。`/vibesubin` でフルスイープ → `write-for-ai` が欠けている `CLAUDE.md` を埋める → `manage-secrets-env` が `.gitignore` とシークレットを監査 → `project-conventions` がブランチ・依存・レイアウトを監査。
- **ゼロから始める**。`manage-secrets-env` が `.env.example` と `.gitignore` を足場組み → `project-conventions` が Dependabot とブランチメモを足場組み → `setup-ci` がワークフローを敷く → `write-for-ai` が README と `CLAUDE.md` を書く。
- **「なんでリポがこんなに重いの?」**。`manage-assets` が肥大化レポートを出す → `refactor-verify` が履歴書き換えや LFS 移行を検証付きで実行。
- **「ページごとに微妙に見た目が違う」**。`unify-design` がトークンファイルを足場組み(無ければ)してドリフトを監査し、コンポーネントをトークン参照に書き戻す → `refactor-verify` が複数ファイルの統合を処理。
- **「編集が終わったので Codex で最終チェック」(Claude Code と Codex プラグイン専用)**。`codex-fix` が現在のブランチに `/codex:rescue` を走らせる → findings を `refactor-verify` の review-driven fix mode にハンドオフ → トリアージ、検証、back-reference コミット。他のホストや、Codex 以外のレビューソース(PR レビュー、Sentry、スキャナー、貼り付けたメモ)の場合は `/refactor-verify` を findings を直接渡して呼び出してください — エンジンは同じで、入力経路だけが違います。

これを自分で計画する必要はありません。スキル側が次のハンドオフを提案します。

## 自分のスキルを追加する

新しいスキルは `plugins/vibesubin/skills/<skill-name>/` 配下の自己完結したディレクトリです。そこに置いてエージェントを再起動すれば、`/vibesubin` も補完メニューも自動で拾います。

```
plugins/vibesubin/skills/<skill-name>/
├── SKILL.md              # 必須、500 行以下、YAML frontmatter 付き
├── references/           # 任意、深掘り用のドキュメント
├── scripts/              # 任意、実行可能なヘルパー
└── templates/            # 任意、プロジェクトにコピーされるファイル
```

新しいスキルが継承するルール: *完了* とは検証済みのことで、主張ではない。`SKILL.md` は 500 行以下で、深い内容は `references/` に。出力は 4 部構成に従う。スイープ対象のスキルは read-only モードが必要。そして、各スキルは自分の自信がどこで切れるかを明示する。

メインプラグインへの同梱を希望するスキルがある? トリガーフレーズ、やること、具体例を添えて [issue を立ててください](https://github.com/subinium/vibesubin/issues)。詳細は [`CONTRIBUTING.md`](./CONTRIBUTING.md) に。

## 哲学

全スキルが従うルールをいくつか。覚える必要はありません — 現在と未来のスキルの一貫性を保つためにあります。

**AI は善意のジュニア開発者です**。ジュニアは一生懸命働きますが、止まって聞くべきタイミングでも先に進んでしまうことがあります。このプラグインは、慎重な手なら差し込んだはずの *「一旦止まって聞く」* モーメントを差し込みます。

**Done は証明であって、主張ではありません**。タスクが complete と言うなら、その裏には実行結果があります — 通ったテスト、一致したハッシュ、生きた `200 OK`。証拠のない主張はバグです。

**ドキュメントは次の AI セッションのためのもので、人間のためだけではありません**。現在の会話はセッションが終われば蒸発します。README、コミット、PR 本文は、新鮮なセッションがコンテキストを再構築できるように書きます。

**既存の conventions がデフォルトです**。気まぐれであなたのブランチ戦略、ファイル配置、config を勝手に書き換えたりはしません。

**ドキュメントは誇張なく、客観的に**。ベンチマークや実際の利用事例なしに *「fast」*、*「production-ready」*、*「best-in-class」* のような語を使いません。能力主張には検証コマンドが並びます。そうでなければ、その文自体を削除します。

このプラグインは [積極的にメンテナンスされています](./MAINTENANCE.md)。リファクタリングツールは進化し、スキルシステムは変わり、LLM の失敗モードはシフトします。各スキルは定期的にレビューされます。

# False-positive triage — a short guide

Most security scanner output is noise. The whole point of `audit-security` is to hand the operator a short list of *real* things. That means every finding has to be triaged before it goes into the report.

This guide gives you one rule per category for how to decide whether a hit is real or a false alarm. Use it, don't memorize it.

---

## Golden rule

Every hit is either:

- **Real** — the operator should fix this
- **False positive** — the code is actually safe, and you can explain *why* in one sentence
- **Needs review** — you can't tell without more context, so ask the operator one specific question

If you can't put a hit into one of those three buckets, you didn't look at it carefully enough.

---

## Per-category rules

### Secrets

- **Real:** any file matching `.env`, `*.pem`, `id_rsa`, `credentials*` in `git ls-files`. Always. Treat as real even if "the repo is private."
- **False positive:** a high-entropy string that is actually a UUID, a git commit SHA, a hash of test data, or a value in a test fixture.
- **Quick check:** is this file in a test directory? Is the value used in a mock? Is it the literal string `change-me` / `placeholder` / `__REPLACE_ME__`? Those are false positives.

### SQL injection

- **Real:** user input (`request.*`, `req.*`, `params`, `body`, query string) flowing into a string that is concatenated or interpolated into a query.
- **False positive:** the interpolated value is a literal constant, a whitelist-checked enum, or an integer produced by `int(...)` / `parseInt(...)`.
- **Quick check:** trace the value back to its source. If the source is anything other than untrusted user input, it's a false positive. Write the source location in your explanation: *"`page` on line 391 is the result of `int(request.query.get('page'))`, so the f-string is safe."*

### Shell injection

- **Real:** user input flowing into `shell=True`, `exec(...)`, backticks, or `$(...)`.
- **False positive:** the argument is a hardcoded path, a whitelist, or a flag (e.g., `subprocess.run(["git", "log", "-5"])` is fine even though it calls git).
- **Quick check:** argument arrays (`["git", "log"]`) are almost always safe. Shell strings (`"git log | ..."`) are almost always suspect.

### Path traversal

- **Real:** user input becomes a file path without whitelist or path-normalization check.
- **False positive:** the path is built from a hardcoded prefix and a whitelisted leaf name (e.g., `os.path.join(ASSETS_DIR, validated_name)` where `validated_name` matched `^[a-z]+$`).
- **Quick check:** is there a regex check *before* the file operation? If yes and the regex is tight, it's a false positive.

### XSS

- **Real:** user input flows into `innerHTML`, `dangerouslySetInnerHTML`, `v-html`, `mark_safe`, `|safe`, or a Markdown renderer whose output isn't sanitized.
- **False positive:** the value is a hardcoded string, a number formatted to a string, or comes from a trusted source (a translation file, a config constant).
- **Quick check:** same question as SQL injection — trace the value to its source.

### Dangerous deserialization

- **Real:** `pickle.loads`, `yaml.load`, `Marshal.load`, `ObjectInputStream`, `unserialize` on anything that could come from a network or from user-controlled files.
- **False positive:** the input is always generated by the same program's own serializer (e.g., a pickle of internal cache state that's only ever read back by the same process, on the same machine).
- **Quick check:** does the input cross a trust boundary (network, filesystem shared with other users, uploaded file)? If yes, it's real. If the serializer and deserializer are the same process on the same disk, it's a false positive.

### Cookie flags

- **Real:** a `set_cookie` call without `httpOnly`, `Secure`, or `SameSite`.
- **False positive:** the framework sets these as defaults and the call is using the default (e.g., Django's `SESSION_COOKIE_HTTPONLY = True` in settings).
- **Quick check:** search the project config for cookie defaults. If they're set globally, the individual call doesn't need to repeat them.

### Wildcard CORS

- **Real:** `Allow-Origin: *` on any endpoint that requires authentication or returns user-specific data.
- **False positive:** a genuinely public endpoint (a static asset CDN, an unauthenticated health check).
- **Quick check:** does the endpoint check a session or an API token? If yes, the wildcard is real. If no, it's usually fine.

---

## When in doubt

Mark it **needs review** and ask the operator **one precise question**:

- Not: *"Is this safe?"*
- Yes: *"On line 87 of `users.py`, the query is built with an f-string that includes `user_id`. Where does `user_id` come from? If it's always an integer from `int(request.query['id'])`, this is safe and I'll mark it as a false positive."*

Precise questions are cheap to answer. Vague questions make the operator silently give up.

---

## Triage output format

The triage must show up in the final report in exactly this structure. The operator reads down the list and trusts the classification because the reasoning is explicit.

```markdown
## Critical (2)

### 1. Secret — .env committed
**What was found:** `.env` is tracked in git (line in `git ls-files`)
**Why it's real:** `.env` files are never safe to commit; always treat as a leak
**Fix:** Remove from git history, rotate every credential inside, add to `.gitignore`

### 2. SQL injection — src/api/search.py:42
**What was found:** `db.execute(f"SELECT * FROM users WHERE name = '{q}'")`
**Why it's real:** `q` comes from `request.query['q']` on line 38, no validation
**Fix:** Use a parameterized query: `db.execute("SELECT * FROM users WHERE name = %s", (q,))`

## False positives (3)

### 1. SQL injection — src/api/list.py:91
**Why not a vulnerability:** The interpolated value is `int(request.query.get('page', '1'))` — an integer, not a string, so the f-string is safe. SQL injection requires string-level control.

...
```

Every false positive gets an explicit *why*. Without the explanation, the operator can't trust that you looked.
